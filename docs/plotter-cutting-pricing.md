# Плоттерная резка: расчёт и термины

Документ описывает учёт **подачи материала** (рулон/лист для печати и списания) и **пробега ножа** (оценка длины траектории резки для тарифа `per_meter`), поля шаблона подтипа и дефолты полей.

## Термины

| Термин | Смысл |
|--------|--------|
| Подача / метраж рулона | Длина проката носителя: для рулона `min(trim_w, trim_h) / 1000 × тираж` (м), как в упрощённом калькуляторе для рулонной печати; при включённом режиме плоттера **roll** без рулонной печати метраж материала считается так же. |
| Пробег ножа `knifePathM` | Оценка погонных метров хода ножа (модель v1: периметр ячеек + «улицы» сетки), см. `plotterLayout.ts`. |
| Листовой плоттер (`sheet`) | Типовой носитель — **SRA3 (320×450 мм)**. В расчёте используются размеры листа материала (`sheet_width` × `sheet_height`); если не заданы — подставляется SRA3 и выводится предупреждение. |
| `meter_basis` в тарифе плоттера (`plotter_cutting_mode_tariffs`) | `knife_path` — тариф за п.м. по пробегу ножа; `feed` — за п.м. по подаче материала. |

## Дефолты полей (мм)

| Режим | Отступ от края носителя (с каждой стороны) | Зазор между этикетками |
|-------|--------------------------------------------|-------------------------|
| Рулон (`roll`) | 10 | 2 |
| Лист (`sheet`) | 15 | 4 |

Типовой формат листа для листового плоттера в производстве — **SRA3 (320×450 мм)**; см. константу `SHEET_PLOTTER_SRA3_MM` в `plotterLayout.ts`.

Переопределение: `cut_margin_mm` / `cut_gap_mm` у строки размера в шаблоне — через `resolvePlotterMargins` в `plotterLayout.ts`.

## Шаблон продукта

- В подтипе (`typeConfigs[typeId].plotter`): `enabled`, `mode` (`sheet` \| `roll`) — какой из двух глобальных тарифов брать; опционально `roll_allowed_material_ids` и `mounting_film_material_id`.
- В корне `simplified`: `print_optional` — допускает расчёт без печати (только материал и плоттер).
- Калькулятор передаёт в конфигурацию `plotter_weeding`, `plotter_mounting` (через `specs`).
- Ставки выборки и накатки задаются в рулонном тарифе **диапазонами по тиражу заказа (шт)**: `weeding_tiers` / `mounting_tiers` (JSON в колонках `weeding_tiers_json`, `mounting_tiers_json`), по тому же принципу строк, что объёмная сетка в проекте. Поля `weeding_price_per_item` и `mounting_price_per_item` дублируют ставку «от 1 шт» для совместимости. При пустых таблицах в расчёт не попадает ненулевая сумма до появления порогов.
- В ответе `GET …/products/:id/schema`, если хотя бы у одного подтипа включён плоттер в режиме **roll**, в `fields` автоматически добавляются два булевых поля с понятными `label` (если таких строк ещё нет в `product_parameters`). В UI калькулятора эти чекбоксы скрываются для подтипа с листовым плоттером или без плоттера.

## Уровни резки (рулон)

Для режима **roll** в тарифе можно задать `cut_level_rules`: список пар «максимальная длинная сторона ячейки (мм) → множитель к ставке за п.м.». Длинная сторона ячейки в расчёте: `max(trim_w, trim_h) + 2×bleed` (мм). Правила сортируются по порогу; применяется первое, где размер ≤ порога (см. `plotterCutLevel.ts`). Настройка в админке — блок «Уровни резки» у рулонного тарифа.

Для листового режима (`sheet`) поле не используется.

## Тарифы и БД

- **Плоттер — отдельная сущность**, не `post_processing_services`: таблица `plotter_cutting_mode_tariffs`, ровно два режима `roll` и `sheet` (рулонный / листовой). Настройка в админке: «Плоттерная резка». API: `GET/PUT /api/pricing/plotter-cutting-tariffs`.
- В расчёте основная резка подставляется по режиму подтипа (`plotter.mode`), с объёмными ступенями по оси тарифа; выборка и накатка — строки **`per_item` по тиражу заказа (шт)** из тех же тарифных таблиц.
- Для совместимости в старых БД могут остаться строки `plotter_cut` в `post_processing_services`; калькулятор **не использует** их для основной резки — только таблицу тарифов выше.
