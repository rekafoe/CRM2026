# Безопасность CRM (операционная модель)

Цель: снизить риск утечки персональных данных и файла **SQLite** (и резервных копий) третьим лицам. Это **не** юрзаключение; по **152-ФЗ** / политике ПДн оформляйте отдельно (ПДн, сроки хранения, ответственный, регистр операций и т.д.).

## 1. База данных (файл на диске)

- **Не** кладите `data.db` и **не** настраивайте `UPLOADS_DIR` / статику на один каталог с БД: загрузки отдаются через тот же процесс, что и API; middleware блокирует расширения вроде `.db`, `.sqlite`, но **лучшая** защита — **изоляция путей** (на Railway: **Volume** только для `DB_FILE`, отдельный путь для `UPLOADS_DIR`).
- **Резервные копии** храните **вне** публичного веб-пространства, шифруйте, ограничьте доступ (учётка, 2FA на облаке), **не** коммитьте в git.
- **Переменные** `DB_FILE`, пароли, `JWT_SECRET`, API-ключи — только **секреты** хостинга (Railway Variables и т.д.), **не** в репозитории.

## 2. API и сеть

- Админ-функции — по **токену** (`Authorization: Bearer`) и, где заложено, ролям (**admin**).
- Публичные эндпоинты (калькулятор, заказ с сайта) — **минимальный** набор; ключ сайта `WEBSITE_ORDER_API_KEY` в production **обязателен** (см. `assertProductionSecurityEnv` в `index.ts`).
- **CORS** — только доверенные `origin` (`CORS_ORIGIN`); не включайте `*` в проде.
- В **проде** за reverse proxy: **TLS** (HTTPS), при необходимости ограничение IP / VPN к админке на уровне балансера.
- **Лимиты** запросов: см. `rateLimiter` (общий и на `/api/auth/login`).

## 3. Заголовки и статика (реализовано в коде)

- **helmet**: базовые HTTP-заголовки, `CSP` для JSON-API **отключён**, чтобы не ломать `/api` и Swagger.
- Перед `express.static` для `/uploads` и `/api/uploads`: **запрет** путей с `..`, расширений **`.db`**, **`.sql`**, **`.dump`**, имён вроде **`data.db`**, вспомогательных файлов SQLite **`-wal` / `-shm`**. Заказные файлы по-прежнему через отдельные **авторизованные** download-роуты, не из произвольного пути.

## 4. Пароли и токены

- Смените **дефолтный** `JWT_SECRET` в production.
- Периодически **ротируйте** `api_token` пользователей / ключ сайта при утечке.
- **Не** логируйте тела запросов с паролями в ясном виде.

## 5. Клиент (фронт)

- В **Vite** в билд попадает только `VITE_*` — **не** кладите туда секреты БД; только **URL API** (публичный).
- **localStorage** с токеном CRM: риск при XSS; снижайте XSS (экранирование, доверенные библиотеки), в перспективе — **httpOnly** cookies (нужен отдельный дизайн сессий).

## 6. Персональные данные (внутренний дисциплинарный минимум)

- Собирайте на сайте/в CRM **только** то, что нужно для оказания услуги.
- **Политика конфиденциальности** и **согласие** на ПДн — публикуйте на сайте; текст согласуйте с юристом.
- Письма (SMTP) — передача email на внешний сервер: учитывайте **оферту/ДПА** провайдера (Яндекс, Mail, и т.д.).

## 7. Реагирование на инцидент

- Смена `JWT_SECRET` и `WEBSITE_ORDER_API_KEY`, сброс всех `api_token` в БД при полной компрометации.
- Уведомление субъектов/регулятора — по **требованию закона** и внутренней политике.

*Обновляйте этот перечень при смене архитектуры (отдельный БД-сервер, S3, SSO и т.д.).*
