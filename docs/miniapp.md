# Mini App (Telegram MAP)

Публичная витрина: `GET /miniapp` — HTML-оболочка, которая подключает внешний bundle Mini App (см. `backend/src/utils/miniappIndexHtml.ts` и `frontend/src/miniapp/`).

## Сессия и заказы

- `POST /api/miniapp/auth` — `initData` от Telegram, ответ: JWT.
- `GET /api/miniapp/me` — профиль.
- `POST /api/miniapp/orders/draft` — создаёт **черновик** заказа. Тело JSON: `{ customer, order: { items, order_notes?, design_help_requested? } }`.  
  `order.items[]`: `type` (как на сайте, часто id продукта), `params` (объект), `price`, `quantity`, `priceType` (например `standard`).  
  `design_help_requested: true` — клиент отметил, что макетов нет и нужна помощь с дизайном; в заметки заказа добавляется пояснение, файлы к позициям не обязательны.  
  Ответ `201`: в теле `order` и `itemIds: number[]` — id строк `items` **в том же порядке**, что и `order.items` (для привязки макетов).
- `POST /api/miniapp/orders/:orderId/finalize` — финализирует черновик.
  Если `design_help_requested !== true`, backend проверяет, что для каждой позиции уже есть прикреплённый макет, и только потом выполняет финальное списание материалов.

## Параметры позиции для CRM

В `items.params` в Mini App рекомендуется передавать:

- `productName` — отображаемое имя;
- `description` — одна строка для списков;
- `parameterSummary` — `Array<{ label: string, value: string }>` (чипы в карточке позиции в CRM);
- `source: 'miniapp'`;
- для калькулятора: `calculator: true`, `configuration` — payload расчёта.

См. также [website-orders-integration.md](./website-orders-integration.md) (аналогия с заказом с сайта).

## Начальный экран и макеты

- По умолчанию открывается **каталог** (вкладка «Каталог» первая в нижней навигации).
- Файл макета выбирается в **калькуляторе до добавления позиции в корзину** и затем хранится в клиентском state до отправки.
- Если не отмечено «Нет макета — требуется помощь с разработкой», макет обязателен **для каждой позиции**.
- На экране оформления Mini App работает как пошаговый flow: **создание черновика -> загрузка макетов -> финализация**.
- При `design_help` сумма подписывается как **цена печати** + текст о расчёте дизайна по телефону, а финализация допускается без файлов.

## Черновик и финализация

Текущий контракт сделан атомарным:

1. клиент отправляет `POST /api/miniapp/orders/draft`;
2. получает `order.id` и `itemIds[]`;
3. загружает макеты к строкам заказа;
4. вызывает `POST /api/miniapp/orders/:orderId/finalize`.

Если загрузка файлов или финализация падает, заказ остаётся в состоянии `draft` и может быть дозавершён повторной отправкой из Mini App. Корзина очищается только после успешного `finalize`.

Draft-заказы не показываются в истории клиента (`GET /api/miniapp/orders`) и не открываются через detail до финализации.

## Загрузка макетов к черновику

После **успешного** `POST /api/miniapp/orders/draft` клиент сопоставляет выбранные в корзине файлы с позициями по **индексу** (`itemIds[i]` ↔ `i`-я строка корзины) и последовательно вызывает:

- `POST /api/miniapp/orders/:orderId/files` — `multipart/form-data`: поле `file`, при необходимости **`orderItemId`** (id из `itemIds` для этой позиции; не передавать = файл «на весь заказ»).

Ограничения: до 25 МБ на файл (и общие `UPLOAD_*` в окружении). `orderItemId` валидируется строго: файл можно привязать только к позиции этого заказа. После загрузки всех обязательных файлов клиент вызывает `finalize`.
