# Оптимизация модуля «Варианты услуги»

## Текущие проблемы
- **ServiceVariantsTable.tsx** — крупный файл: таблица, тулбар, автосохранение, синхронизация с сервером, модалка диапазонов.
- Глубокая вложенность: `typeNames.map` → `level1` → `level2` с повторяющимися колбэками.
- Несколько хуков в одном компоненте: варианты, редактирование, модалка, операции, локальные изменения.
- Два слоя состояния: `serverVariants` + локальные черновики и очереди — сложнее отлаживать гонки (автосохранение / reload / ввод).
- Иерархия в основном в **JSON `parameters`** — на фронте нужны эвристики группировки; риск расхождений «в БД есть, в таблице не видно».

## 3. Один хук «варианты + сохранение» (предложение)
**Файл:** `hooks/useVariantsTable.ts`

**Объединить:** `useServiceVariants` + `useLocalRangeChanges` + логику из `saveChangesToServer` (сейчас размазана по `ServiceVariantsTable`).

**Предлагаемый контракт на выходе:**

| Поле / метод | Назначение |
|--------------|------------|
| `variants` | То, что показываем (после merge с черновиком или с сервера) |
| `groupedVariants`, `typeNames` | Результат `groupVariantsByType` + ключи |
| `pendingChanges`, `hasUnsavedChanges` | Как сейчас |
| `save`, `cancel` | Явное сохранение / сброс |
| `autosave` | Опционально: `{ enabled, debounceMs, schedule, flush }` внутри хука |
| `syncState` | Удобочитаемое: `'idle' \| 'dirty' \| 'saving' \| 'error'` |
| `operations` | Обёртка над `useVariantOperations` или суженный API |
| `reload`, `invalidateTiersCache` | Как сейчас, но из одного места |

**Эффект:** одна точка входа, проще тесты и сценарии «сохранилось / ошибка / конфликт».

## 8. Сеть и бэкенд: batch-сохранение (предложение)
**Проблема:** сейчас сохранение — цепочка запросов (создание/обновление вариантов, диапазоны, цены). Много round-trip, сложный откат при ошибке на середине.

**Предложение:** endpoint в духе:

```http
PUT /api/pricing/services/:serviceId/variants/sync
```

**Тело (черновик контракта):**

```ts
{
  variants: {
    create: Array<{ clientTempId?: string; variantName: string; parameters: object; sortOrder?: number }>;
    update: Array<{ id: number; variantName?: string; parameters?: object; ... }>;
    delete: number[];
  };
  ranges?: { add?: number[]; remove?: number[]; edit?: Array<{ index: number; minQuantity: number }> };
  prices?: Array<{ variantId: number; minQuantity: number; price: number }>;
}
```

**Ответ:** полный снимок вариантов + tiers (или id-маппинг `clientTempId → serverId` для новых строк), **одна транзакция** на бэкенде.

**Эффект:** атомарность, меньше запросов, проще «всё или ничего».

## 9. Данные: нормализация иерархии в БД (долгосрочно)
**Сейчас:** `parentVariantId`, `type`, `subType` живут в `parameters` (JSON).

**Предложение:** явные колонки, например:
- `parent_variant_id` (nullable FK на тот же `service_variants`),
- при необходимости `tree_sort` / `level` для сортировки и запросов.

**Эффект:** группировка на фронте без эвристик «кто корень»; проще отчёты и валидация на сервере.
